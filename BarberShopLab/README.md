## 实验结果分析

### 三种方式对比

**代码复杂度与可维护性**

1. Monitor：代码最短，状态少，维护成本低。
2. Semaphore：中等复杂度，需要理解 permit 计数语义。
3. ReentrantLock：复杂度最高，必须自行维护条件与状态一致性（如 ticket 机制）。

**正确性风险点**

1. Monitor：主要风险是 if(wait) 写成一次判断，未用 while 防伪唤醒。
2. Semaphore：主要风险是 acquire/release 配对不当导致死锁或“多放行”。
3. ReentrantLock：主要风险是 signal 时机和等待条件设计不严谨，导致丢信号/误唤醒后卡死。
	1. 本项目修复前的 ReentrantLock 就是典型案例：用全局 waiting 推断“自己是否被叫号”不可靠。后来改为每个用户拿一个号，通过这个号的比较来判断是否被叫到，防止顾客一直沉睡

**性能与扩展性**

1. Monitor：单锁串行临界区，性能通常够用。
2. Semaphore：在生产者-消费者式问题上表达直接，扩展多资源时清晰。
3. ReentrantLock：功能最强（可中断、公平锁、多 Condition），适合复杂协作场景。

### 状态机说明

**统一问题模型**

1. 共同状态可抽象为：`Open/Running`（营业中）、`BarberSleeping`（无顾客阻塞等待）、`Serving`（理发中）、`Closing`（停止接客并清场）、`Closed`（理发师退出）。
2. 共同事件：`CustomerArrive`、`SeatAvailable`、`SeatFull`、`BarberPick`、`CloseShop`、`NoWaitingAndClosing`。

**Monitor（`synchronized + wait/notifyAll`）**
1. 状态变量：`waiting, served, dropped, closing`（由对象监视器互斥保护）。
2. 顾客状态机：
1. `Arrive` -> 判断 `waiting >= chairs`。
2. `SeatFull` -> `dropped++` -> `Leave`（终态）。
3. `SeatAvailable` -> `waiting++` -> `notifyAll()` -> `Done`（顾客线程结束，不阻塞等理发）。
3. 理发师状态机：
1. `LoopStart` -> 若 `waiting==0 && !closing` -> `wait()` -> `BarberSleeping`。
2. 被唤醒后重检条件（`while`）-> 若 `waiting==0 && closing` -> `Exit`（终态）。
3. 否则 `waiting--, served++` -> `Serving` -> 返回外层睡眠模拟理发 -> 回到 `LoopStart`。
4. 关店迁移：
1. `CloseShop`：`closing=true; notifyAll()`。
2. 理发师在“无人且closing=true”时从循环退出进入 `Closed`。

**Semaphore（`mutex/customers/barbers`）**
1. 状态变量：`waiting, served, dropped, closing`，以及信号量计数 `customersPermits/barbersPermits`。
2. 顾客状态机：
1. `Arrive` -> `mutex.acquire()`。
2. `SeatFull` -> `dropped++` -> `mutex.release()` -> `Leave`。
3. `SeatAvailable` -> `waiting++` -> `mutex.release()` -> `customers.release()`（投递“有顾客”事件）。
4. `barbers.acquire()`（等待叫号）-> `StartHaircut` -> `Done`。
3. 理发师状态机：
1. `LoopStart` -> 若 `closing && customers.availablePermits()==0` -> `Exit`。
2. 否则 `customers.acquire()`（无顾客即阻塞睡眠）。
3. `mutex.acquire()` -> 若 `waiting>0` 则 `waiting--, served++` -> `mutex.release()`。
4. `barbers.release()`（放行一位顾客）-> `Serving` -> 回 `LoopStart`。
4. 关店迁移：
1. `CloseShop`：`closing=true; customers.release()`（防止理发师永久阻塞）。
2. 理发师下一轮检测到“closing 且无顾客令牌”后退出。

**ReentrantLock（`Condition`，修复后票号版）**

1. 状态变量：`waiting, served, dropped, closing, nextTicket, servingTicket`。
2. 顾客状态机：
1. `Arrive` -> `lock.lock()`。
2. `SeatFull` -> `dropped++` -> `unlock` -> `Leave`。
3. `SeatAvailable` -> `waiting++`，分配 `myTicket=nextTicket++`，`customerArrived.signal()`。
4. 进入等待：`while (myTicket != servingTicket && !closing) barberReady.await()`。
5. 条件满足（轮到自己）-> `StartHaircut` -> `unlock` -> `Done`。
3. 理发师状态机：
1. `LoopStart` -> `while (waiting==0 && !closing) customerArrived.await()`（睡眠）。
2. 若 `waiting==0 && closing` -> `Exit`。
3. 否则 `waiting--, served++`，并 `servingTicket++`。
4. `barberReady.signalAll()`（唤醒全部顾客重检票号）-> `Serving` -> 回 `LoopStart`。
4. 关店迁移：
1. `CloseShop`：`closing=true; customerArrived.signalAll(); barberReady.signalAll()`。
2. 所有等待线程被唤醒并依据条件退出，理发师最终进入 `Closed`。
